# Recommended Answer (English)

One of the most interesting incidents I worked on involved storage performance in a GPU infrastructure environment supporting AI workloads.

The environment relied on VAST Storage and high-performance networking to provide large-scale data access for GPU training workloads.

Users reported that storage throughput was significantly lower than expected, which directly affected AI training performance.

Initially, we suspected the storage platform, network infrastructure, or RDMA configuration because these are common causes of throughput issues in GPU environments.

However, after validating storage health, network connectivity, and RDMA-related configurations, all major infrastructure components appeared healthy.

We continued investigating the entire data path and eventually discovered that the issue originated in the operating system layer.

Large sequential IO requests generated by applications were being fragmented by the Ubuntu NFS client into many smaller IO operations.

As a result, IOPS increased dramatically, but throughput remained much lower than expected.

The long-term solution was to upgrade to an operating system version containing the fix.

However, an immediate OS upgrade was not feasible in production.

To address the issue, the VAST engineering team provided a driver enhancement that reassembled fragmented IO requests before transmission.

After applying the fix, throughput returned to expected levels and GPU workload performance improved significantly.

The incident reinforced the importance of validating behavior across every layer of the stack rather than assuming the problem exists where the symptoms appear.

# Situation

I was supporting a GPU infrastructure environment used for AI training workloads.

The environment depended on VAST Storage to provide high-throughput data access for GPU clusters.

Users reported significantly lower throughput than expected.

This directly affected training efficiency and overall workload performance.

---

# Task

My responsibility was to identify where the bottleneck existed and determine whether the issue originated from:

- Storage
- Network
- RDMA
- Host Configuration
- Operating System

The challenge was that all major infrastructure components initially appeared healthy.

---

# Actions

## Step 1

Establish Baseline

We compared performance across multiple environments.

Observed Results

Expected

40+ GB/s

Observed

~9.7 GB/s

The throughput gap was significant.

---

## Step 2

Validate Storage Layer

We reviewed:

- VAST Storage Health
- Storage Metrics
- Storage Configuration
- Backend Performance

No significant issues were identified.

---

## Step 3

Validate Network Layer

We reviewed:

- Network Connectivity
- Throughput
- RDMA Configuration
- RoCE Configuration

Network infrastructure appeared healthy.

---

## Step 4

Investigate End-to-End Data Path

We expanded the investigation beyond storage and networking.

We analyzed:

- Application IO Behavior
- Operating System IO Handling
- NFS Client Behavior

This revealed that large IO requests generated by applications were being fragmented into many smaller IO operations before transmission.

---

## Step 5

Identify Root Cause

The issue was traced to an Ubuntu operating system behavior affecting the NFS client layer.

Large sequential IO operations were being split into smaller requests.

This resulted in:

- Excessive IOPS
- Poor throughput
- Reduced storage efficiency

The storage platform itself was healthy.

The network infrastructure was healthy.

The issue originated from the operating system layer.

# Resolution

## Long-Term Fix

Upgrade to an operating system version containing the fix.

---

## Immediate Production Fix

An immediate OS upgrade was not feasible.

The VAST engineering team provided a driver enhancement.

The modified driver reassembled fragmented IO requests before transmission.

This restored the intended IO pattern and significantly improved throughput.

# Technical Lessons Learned

## Lesson 1

Never assume storage throughput issues originate from the storage platform.

---

## Lesson 2

Performance problems often span multiple layers.

Symptoms may appear in storage while the root cause exists elsewhere.

---

## Lesson 3

Always validate actual IO behavior.

Configuration alone does not guarantee performance.

---

## Lesson 4

A systematic elimination process is essential.

We ruled out:

- Storage
- Network
- RDMA

before identifying the operating system layer.
